PC 启动过程中发生了什么？

发表于 2020-12-06 | 分类于 操作系统 | | 热度: ℃
字数统计: 1.1k 字 | 阅读时长 ≈ 4 分钟

今天重新回顾了一下 828 的 Lab 1，对 PC 的启动过程做一个总结，加深印象。

简而言之

内核启动过程：

  1. 通电，启动BIOS。
  2. BIOS 初始化各种设备，读取磁盘第一个扇区，加载 MBR。
  3. MBR 读取磁盘第二个扇区，加载内核映像，跳转至内核入口。

1 从 BIOS 开始

  1. CPU 通电之后，根据固化在 CPU 内部的指令，从 0x00ffff0 [f000:fff0] 处开始执行。（这个地址是 BIOS 的起始地址）

  2. 此处存储的是一条跳转指令，ljmp $0xf000, $0xe05b。

  3. 意为跳转到 [f000:e05b]，即地址 0x00fe05b 处，开始执行 BIOS 的功能。

  4. 首先，设置一个 IDT，初始化各种设备，比如 PCI 总线。

  5. 然后，寻找一个可启动的设备，硬盘、软盘、CD-ROM 等。

  6. 找到之后，读取第一个扇区，Boot Loader（MBR，通常小于 512B）存储在磁盘的第一个扇区。读取 Boot Loader，并将控制权交给他。

  7. BIOS 执行过程结束。

2 Boot Loader

Boot Loader 被加载到物理地址 0x7c00 处。大小为 512B，从 0x7c00 到 0x7dff 。

Boot Loader 由两个文件组成，一个汇编文件 boot.S，一个 C 源文件 main.c。

功能为从磁盘第二个扇区加载内核到内存中，并跳转到内核第一条入口。

3 加载内核

先执行 boot.S。这个文件实现的功能：转换到32位保护模式，跳转到 main.c 。

  1. 设置内核代码段选择子，数据段选择子，保护模式启动标志（CR0_PE_ON）。
  2. 关中断。设置重要的段寄存器（DS，ES，SS）。
  3. 从实模式切换至保护模式，使用引导GDT和段转换，使虚拟地址与其物理地址相同，因此有效内存映射在切换期间不会发生变化。
  4. 跳转到下一条指令，但是在 32 位代码段，处理器切换到 32 位模式。
  5. 设置保护模式数据段寄存器（PROT_MODE_DSEG，DS，ES，FS，GS，SS）。
  6. 设置栈指针，调用 bootmain()。

之后是 main.c。加载内核映像，加载地址是 0x00100000，然后转向内核入口地址 0x0010000c。

  1. 进入 bootmain() 函数。
  2. 读取磁盘第二个扇区。
  3. 判断是否是一个有效的 ELF。
  4. 加载每一个程序段。
  5. 从 ELF 头进入程序入口点（ELFHDR->e_entry : 0x10018）。
  6. 内核加载完毕，控制权交给内核，加载过程到此为止。

4 i386 平台以及保护模式

i386 处理器第一次引入了 32 位扩展寄存器。

实模式中，内存有 2^20 字节，即 00000~FFFFF。所以需要两个 16 位的寄存器来定位一个地址，具体来说，就是把第一个寄存器当作选择器（selector），第二个寄存器作为段内偏移（offset），通过公式 16 * select + offset 来计算实际地址。

i386 引入了 32 位保护模式，和 i286 的 16 位保护模式主要有两点不同：

  • 段内偏移扩展至 32 位。因此，段的尺寸可以增长至 4GB。
  • 段可以被分成更小的 4K 单元，也就是页。虚拟内存系统使用页而不是段，那么同一时间在内存中的就可以是段的一部分，而不用是整个段。

5 ELF

Executable and Linking Format (ELF)。

内核映像以 ELF 的格式存在。

左边为链接视图，右边为执行视图。

链接视图：

静态链接器（即编译后参与生成最终ELF过程的链接器，如ld ）会以链接视图解析 ELF。编译时生成的 .o（目标文件）以及链接后的 .so （共享库）均可通过链接视图解析，链接视图可以没有段表（如目标文件不会有段表）。

执行视图：

动态链接器（即加载器）会以执行视图解析ELF并动态链接，执行视图可以没有节表。
