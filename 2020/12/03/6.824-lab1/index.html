请随意借用 mrsequential.go 中的代码。观察 mrapps/wc.go 以了解 MapReduce 应用应该长什么样。

Your Job

实现一个分布式 MapReduce 应用，它包含两个程序，master 和 worker。 只有一个主进程和一个或多个工作进程并行执行。在一个真实的系统中，worker 将在一堆不同的机器上运行，但是对于这个 lab 来说，你将在一台机器上运行它们。worker 通过 RPC 和 master 通话。每个 worker 进程将向 master 请求任务，从一个或多个文件中读取任务的输入，执行任务，并将任务的输出写入一个或多个文件。如果一个 worker 没有在合理的时间内完成任务（对于这个实验，用 10 秒），master 应该注意，并把同样的任务交给另一个 worker。

我们给了你一点代码来开始。master 和 worker 的 “main” 程序在 main/mrmaster.go 和 main/mrworker.go，不要更改这些文件。mr/master.go， mr/worker.go，和 mr/rpc.go 中放置你的实现。

A few rules：

  • map 阶段应该将中间 keys 划分成 nReduce 份 reduce 任务，nReduce 是 main/mrmaster.go 传递给 MakeMaster() 的参数。
  • worker 实现应该将第 X 个 reduce 任务的输出结果放在 mr-out-X 文件中。
  • 一个 mr-out-X 文件应该每个 Reduce 函数输出的一行。该行应该以Go “%v %v” 格式生成，用 key 和 value 调用。看一下 main/mrsequential.go ，找一行注释 “this is the correct format”。如果您的实现与这种格式偏离太多，测试脚本就会失败。
  • 你可以修改 mr/worker.go，mr/master.go 和 mr/rpc.go。您可以临时修改其他文件进行测试，但请确保您的代码与原始版本兼容；我们将使用原始版本进行测试。
  • worker 应该将中间 Map 输出放在当前目录的文件中，您的 worker 稍后可以将它们作为 Reduce 任务的输入读取。
  • main/mrmaster.go 期望 mr/master.go 实现一个 Done() 方法，这个方法当 MapReduce 作业完全完成时返回 true；那时， mrmaster.go 将退出。
  • 当作业完全完成时，worker 进程应该退出。实现这一点的一个简单方法是使用call()的返回值：如果 worker 未能联系到 master，则可以假定master 已退出，因为作业已完成，因此 worker 也可以终止。根据您的设计，您可能会发现有一个“please exit”的伪任务，master 可以将它交给 worker。

Hints：

  • 一种方法是修改 mr/worker.go‘ 的 Worker() 向 master 发送 RPC 请求任务。然后修改 master 以使用尚未开始的 Map 任务的文件名进行响应。然后修改 worker 以读取该文件并调用应用程序 Map 函数，如 mrsequential.go所示。

  • 应用程序 Map 和 Reduce 函数在运行时使用 Go-plugin 包从名称以 .so 结尾的文件加载。

  • 如果你改变 mr/目录下的任务东西，你可能需要重新编译你使用 MapReduce 插件，就像 go build -buildmode=plugin ../mrapps/wc.go。

  • 这个 Lab 依靠 worker 共享一个文件系统。当所有的 worker 运行在同一台机器上时，这很简单，但是如果 worker 运行在不同的机器上，则需要像 GFS 这样的全局文件系统。

  • 中间文件的合理命名约定是 mr-X-Y，其中 X 是 Map 任务编号，Y 是 reduce 任务编号。

  • worker 的 map 任务代码将需要一种方法来存储文件中的中间 key/value 对，以便在 reduce 任务期间能够正确地读回。一种可能是使用 Go 的 encoding/json 包。要将 key/value 对写入 JSON 文件，请执行以下操作：

  • worker 的 map 部分可以使用 ihash(key) 函数（worker.go 中）为给定的 key 选择 reduce 任务。

  • 你可以从 mrsequential.go 借用用于读取 Map 输入文件、在 Map 和 Reduce 之间排序中间 key/value 对，以及将 Reduce 输出存储在文件中的代码。

  • master 作为 RPC 服务器将是并发的；不要忘记锁定共享数据。

  • 使用 Go 的竞速检测器，有 Go build-race 和 Go-run-race。test-mr.sh 有一条注释，说明如何为测试启用竞争检测器。

  • worker 有时需要等待，例如，在最后一个 map 完成之前不能开始 reduce。一种可能是 worker 定期向 master 请求工作，在每次请求的等待时间使用 time.Sleep() 进行休眠。另一种可能是 master 中的相关 RPC 处理程序有一个等待的循环时间，使用 time.Sleep() 或 sync.Cond。Go 在它自己的线程中为每个 RPC 运行处理程序，因此一个处理程序正在等待的事实不会阻止 master 处理其他 RPC。

  • master 无法可靠地区分出崩溃的 worker、活着但因某种原因停滞的 worker，以及正在执行但速度太慢而无法发挥作用的 worker。你能做的最好的方法就是让 master 等待一段时间，然后放弃并重新将任务分配给其他 worker。对于这个 lab，让 master 等十秒钟，然后 master 应该假设 worker 已经死了（当然，可能没有）。

  • 要测试崩溃恢复，可以使用 mrapps/crash.go应用程序插件。它在Map和Reduce函数中随机退出。

  • 为了确保没有人在崩溃的情况下观察到部分写入的文件，MapReduce 论文提到了使用临时文件并在完全写入后自动重命名的技巧。你可以用ioutil.TempFile 文件创建临时文件并os.Rename以原子方式重命名它。

  • test-mr.sh 运行 mr tmp 子目录中的所有进程，因此，如果出现问题，需要查看中间或输出文件，请查看那里。
